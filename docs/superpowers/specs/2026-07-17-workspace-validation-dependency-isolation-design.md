# Workspace 校验依赖隔离设计

## 背景与根因

桌面编辑器执行 `target.go` 校验时，会把 workspace 中所有 `.obpf` 递归复制到临时沙箱。Go 引擎初始化会先解析并编译沙箱内的全部函数，因此任意无关函数的解析错误、未注册节点或签名错误都会被错误地显示为当前蓝图的编译失败。

这属于校验发现范围错误，不是当前蓝图内容错误。目标诊断仍属于 `target.go`，只能设置 `blocksRun`，不能设置 `blocksSave`。

## 方案选择

采用“函数依赖闭包”方案：扫描 workspace 只建立轻量函数索引，不把所有函数直接交给引擎；从当前文档的 `origin.function.call` 和 `origin.timer.set-by-function` 开始，按 `functionId`、`functionName`、函数自身 `functionId`、`graphName` 和 workspace 相对路径解析依赖，递归复制实际引用到的 `.obpf`。

没有采用以下方案：

- 硬编码排除 `examples/`：可能屏蔽用户合法项目结构，且不能解决其他目录中的无关坏函数。
- 仅复制 `functions/`：限制了现有 workspace 可以递归放置函数的兼容语义。
- 修改 Go 引擎全局加载器：运行时仍需要完整加载部署目录；问题只存在于编辑器临时校验沙箱，不应改变生产加载契约。

## 数据流与边界

1. 当前编辑内容始终写入沙箱，磁盘上的同路径旧内容不会覆盖它。
2. workspace 扫描收集每个 `.obpf` 的相对路径，并尽力读取 `functionId`、`graphName` 和节点引用。
3. 当前文档中的每个函数引用作为依赖根；选中的函数继续提供下一层引用，直到闭包稳定。
4. 同一别名若有多个所有者，全部复制，让现有引擎继续报告真实的函数键冲突。
5. 无法解析的无关函数不会进入沙箱；若通过相对路径被实际引用，则仍复制并由引擎报告其解析错误。
6. 依赖无法解析时不伪造成功：调用节点仍由现有引擎报告“函数不存在”。

## 诊断显示

当引擎返回结构化 `BlueprintError` 时，复制错误对象并把临时 `SourcePath` 替换为 workspace 原始路径，再生成消息。问题面板的 `SourcePath` 与消息都不得暴露 `origin-blueprint-validation-*` 临时目录。workspace 依赖错误继续带 `Workspace function:` 前缀。

## 兼容性与安全性

- 不修改 `.vgf/.obp/.obpf` 持久化格式。
- 不修改核心图分析器、`blocksSave` 或严格保存门禁。
- 不修改 Go 引擎生产目录加载语义。
- 使用迭代队列计算依赖闭包，避免深层函数链递归导致调用栈风险。
- 文件只复制到现有临时目录；相对路径必须保持在 workspace 根目录内。

## 验收标准

- 当前有效图不再因 workspace 中无关的坏 `.obpf` 报 `engine.*` 错误。
- 当前图直接或传递引用的坏函数仍然报错。
- 重复函数别名仍然报冲突。
- 错误来源和消息显示真实 workspace 路径，不显示临时沙箱路径。
- 所有 Go 测试、race 测试、前端测试与构建、Wails 构建通过。
