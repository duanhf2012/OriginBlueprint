# P1 行为测试收尾设计

## 目标

在不进入第二批“编辑器校验与真实引擎编译规则统一”、不修改 `.vgf` 迁移语义的前提下，为已经实现的四项 P1 防护补齐可执行的行为回归测试，并恢复项目主验证命令的可信度。

本批完成后，以下结论必须由测试行为证明，而不是由源码字符串匹配推断：

1. 重复打开已存在的脏标签不会替换文档、路径或 `dirty` 状态。
2. 无法恢复的节点、连线和被规范化的持久化属性会触发源文件覆盖保护。
3. 恢复副本、取消、强制覆盖三种保存决策产生不同且正确的持久化动作；恢复异常时不允许强制覆盖。
4. Sequence 的第 13 至 256 个输出能够保存、恢复并保留对应连线。

## 非目标

- 不统一桌面校验器与 Go VM 编译器。
- 不新增自动保存。
- 不改变 `.vgf` 导入、导出、legacy 隐藏状态或扩展字段语义。
- 不重构 Undo/Redo。
- 不搭建完整的浏览器端到端或 Wails GUI 自动化框架。
- 不通过放宽断言、跳过样例或删除测试来使 `go test ./...` 变绿。

## 推荐结构

### 1. 可测试的打开与保存策略

从 `App.vue` 中提取无 UI 依赖的策略模块 `frontend/src/documentSafety.ts`。模块只处理普通对象和明确的动作类型，不访问 Vue、Rete、Wails 或浏览器对话框。

模块包含：

- 路径归一化和已打开标签查找。
- 恢复报告是否构成兼容性损失的判断。
- 根据 `saveAs`、源路径、恢复报告、恢复异常和格式能力计算保存选项。
- 对 `copy`、`force`、`cancel` 的最终持久化动作建模。

`App.vue` 继续负责显示对话框和调用平台 API，但必须消费策略模块的返回值。测试因此验证生产代码实际使用的决策逻辑，而不是复制一份测试专用规则。

### 2. 可测试的恢复预检

在创建 Rete 节点前生成恢复计划。恢复计划至少包含：

- 可恢复节点列表。
- 丢失节点及原因。
- 可恢复连线列表。
- 丢失连线及原因。
- 被规范化节点属性及原值、恢复值。

恢复计划依赖 `RestoreCapabilities`：它提供已知 `typeId` 判断，以及按节点快照返回输入、输出端口键集合的函数。实际 `restore` 使用该计划创建 Rete 对象，测试使用伪能力描述验证未知节点、缺失端口、悬空端点和动态 Sequence 输出。

该边界避免在本批引入 DOM、Canvas 或完整 Rete 挂载测试，同时确保“判定哪些内容会丢失”的核心行为由真实数据驱动。

### 3. Sequence 行为

动态输出规范化保留以下契约：

- `0` 继续按兼容规则恢复为默认值 `3`。
- 合法范围为 `1..256`。
- 非整数、负数、非有限值和大于 `256` 的值会被规范化，并进入 `alteredNodes`。
- 合法的 `then12`、`then255` 端口及连线必须进入恢复计划并通过快照 round-trip。

## 测试方案

### TypeScript 行为测试

采用与现有 Vite 3 兼容的 Vitest `0.34.x`，测试环境使用 Node，不引入 jsdom、Vue Test Utils 或浏览器自动化依赖。行为测试直接导入纯 TypeScript 策略模块，命令固定为 `vitest run tests/p1Behavior.test.ts`，并接入现有 `test:layout`。依赖变更只允许新增 Vitest 及其锁文件解析结果；如果安装导致无关依赖升级，回退无关升级后再继续。

必须覆盖：

1. 同一路径不同斜杠、Windows 大小写差异能够命中已有标签。
2. 命中已有脏标签后返回 `focus-existing`，原文档引用和 `dirty` 保持不变。
3. 未知节点被列入 `droppedNodes`，依赖该节点的连线被列入 `droppedConnections`。
4. 已知节点上的失效输入或输出端口被列入 `droppedConnections`。
5. legacy opaque state 未进入可视节点列表时，不被误报为恢复丢失。
6. 普通保存遇到恢复损失时默认返回 `recovery-copy`。
7. 用户取消时不调用任何持久化动作。
8. 强制覆盖仅在非 fatal 且格式允许时可用，并返回 `force-source-with-backup`。
9. fatal 恢复只允许恢复副本或取消。
10. Sequence 的 `then12`、`then255` 和对应连线完整 round-trip。

现有 `p1Safety.test.js` 中与上述行为重复的源码字符串断言应删除；仍有价值的公共 API 形态断言可以保留，但不能作为 P1 行为验收依据。

### Go 行为测试

保留并运行现有测试：

- 强制保存先备份原始字节，再替换源文件。
- Sequence 负数与 `257` 返回结构化问题。
- 函数签名超过 `128` 返回结构化问题。
- legacy 端口超过 `4096` 返回结构化问题。
- 20,000 层执行流不会因递归遍历崩溃。

新增或补强失败路径测试时，不依赖操作系统权限技巧；优先通过可控的临时目录和注入式文件操作边界验证“备份失败不替换源文件”。如果为此需要显著重构持久化层，则推迟到 P2，不扩大本批范围。

### 三个既有 Go 样例失败

当前三个失败来自测试把 `build/bin/vgf` 中的原生 `schemaVersion: 1` 文档送入 legacy 迁移入口，或依赖运行时节点 schema 的样例环境。处理原则：

- 先确认每个样例实际是 native 还是 legacy。
- 测试必须走与产品打开流程相同的格式识别入口。
- legacy round-trip 断言只用于真实 legacy 文档。
- 运行时 schema 依赖必须由仓库内固定 fixture 提供，不能依赖外部 sibling 项目。
- 不修改产品迁移逻辑来迎合错误的测试入口。

## 错误处理与状态约束

- 恢复策略的任何异常都必须形成 fatal 状态；fatal 状态不能产生强制覆盖动作。
- 保存成功前不能清除 `restoreLoss`、`restoreFatal` 或 `dirty`。
- 恢复副本成功后可将当前标签切换到副本路径并清除兼容性保护；原始路径不被写入。
- 强制覆盖成功后才能清除保护，并向用户显示 `.bak` 路径。
- 对话框关闭或取消不得改变标签状态。

## 验收命令

在仓库根目录：

```powershell
go test ./...
go test -race ./engine/go/blueprint -count=1
go vet ./...
```

在 `frontend/`：

```powershell
npm run test:layout
npm run build
```

最终验收要求所有命令退出码为 `0`。此外，用一个包含未知节点/失效连线的测试 fixture 和一个包含 `then255` 连线的 native fixture 做桌面人工冒烟检查，确认保存对话框与备份提示符合设计。

## 风险控制

- 策略提取只移动决策，不改变视觉布局、快捷键或文件格式。
- 恢复预检与实际恢复共用同一份计划，避免测试逻辑与生产逻辑分叉。
- 不为测试引入完整浏览器自动化，控制依赖和执行时间。
- `.vgf` fixture 只用于验证现有行为，测试不得重写仓库的大型历史文件。

## 完成定义

只有在行为测试真实调用生产策略、五条验收命令全部通过、两类 fixture 冒烟验证完成后，P1 才标记为关闭。随后才能为第二批“编辑器校验与真实引擎编译规则统一”单独编写设计和实施计划。
