# 蓝图持久化与执行引擎安全修复设计

日期：2026-07-10

## 1. 背景

本设计针对蓝图文件往返转换、Go 执行引擎校验、异步生命周期、热加载和函数局部变量语义进行收敛。修复目标是阻止静默数据丢失、释放后继续执行和非法图进入运行时，同时保持现有蓝图绘制、端口显示、节点 JSON 加载和合法流程环路行为不变。

现状审计已经确认：

- 158 个 legacy `.vgf` 样本中有 14 处未映射 `port_defaultv` 在导入后再导出时丢失。
- legacy 导出遇到无法映射的节点、连接或端口时会跳过，而不是报错。
- `Sleep` 的延迟回调在实例释放后仍可继续执行。
- 外部定时器触发后没有从实例的定时器集合中移除。
- 编译器对重复节点 ID、端口越界、执行/数据端口方向错误和内建端口类型不匹配校验不足。
- legacy 图允许同一执行输出连接多个后继，而当前 Go 引擎只保存最后一个后继；直接增加会话互斥会破坏同步函数 continuation，必须独立设计 activation/frame 调度。
- 热加载只替换编译结果，没有按新定义重建实例变量。
- 函数调用当前会创建新的变量表，这一行为正确；但共享父级变量锁的实现和“函数共享实例变量”的文档描述均不正确。

## 2. 目标

1. legacy `.vgf` 的已知内容必须可往返保存，无法表达的内容必须显式失败，不能静默丢失。
2. 编译阶段拒绝确定无效的图结构，避免运行时端口类型被意外改写。
3. `ReleaseGraph` 标记释放后不能再开始新的执行片段；释放前已经取得执行资格的在途片段允许完成，挂起片段不能在释放后重新取得恢复资格。
4. 对仍存在于新加载集合中的图，热加载后的新执行使用新图定义和一致的新变量状态；进行中的执行不被中途换底。被新集合删除的图按第 8 节保留已有实例。
5. 函数蓝图中的变量是调用帧局部变量，每次调用、递归调用都独立，且与调用方同名变量隔离。
6. 修改保持局部，不触碰 Rete 绘制、节点视觉、节点 schema 解析和现有保存格式选择策略。

## 3. 非目标

- 本轮不统一 `.obp` 与 `.vgf` 的文件语义。
- 本轮不恢复缺失的业务节点定义。
- 本轮不全面禁止执行环路。合法循环节点依赖回边，循环防护应作为独立设计处理。
- 本轮不允许函数读取或写入调用方蓝图变量。
- 本轮不重构同步执行热路径，也不以减少 `clonePorts` 分配为主要目标。
- 本轮不修改多执行输出的运行语义，也不把这种历史合法结构判错。该缺陷需要独立的 activation/frame、终止取消和 continuation 调度设计，不能用单个 `sync.Mutex` 修补。
- 本轮不修改蓝图节点和端口的显示、布局、连线交互。

## 4. Legacy 默认值残留数据

### 4.1 数据结构

在 `GraphLegacyState` 中增加可选残留字段，用可见节点 ID 关联无法映射到当前 schema 输入端口的 legacy 默认值。`port_defaultv` 实际是字符串键 JSON 对象，不是数组，因此残留数据必须保留原始键：

```json
{
  "legacy": {
    "format": "vgf",
    "residualNodeDefaults": {
      "node-17": {
        "class": "DoHitEffect",
        "values": {
          "13": 0
        }
      }
    }
  }
}
```

Go 数据契约：

```go
type GraphLegacyResidualDefaults struct {
    Class  string                 `json:"class"`
    Values map[string]interface{} `json:"values"`
}

type GraphLegacyState struct {
    // 现有字段保持不变。
    ResidualNodeDefaults map[string]GraphLegacyResidualDefaults `json:"residualNodeDefaults,omitempty"`
}
```

TypeScript 的 `GraphLegacyState` 增加同构可选字段。前端继续把 `legacy` 当作不透明持久化数据深拷贝，不把残留默认值渲染成额外端口或控件。

### 4.2 导入规则

1. 根据 legacy class 的当前 schema，把可映射输入默认值写入节点可编辑的 `values`。
2. 只有规范的非负十进制键才参与端口索引映射：解析成功后还必须满足 `strconv.Itoa(index) == rawKey`。
3. 索引超出当前 schema 输入数量、负数键、非数字键和非规范数字键，都按原始字符串键写入 `residualNodeDefaults[nodeID].values`。
4. 不可映射键不得因 `strconv.Atoi` 失败而误写到输入端口 `0`。
5. 仅当一个节点确实存在残留值时创建对应条目。
6. 残留条目同时记录原始 legacy class，防止节点类型变化后错误套用旧值。正常编辑流程使用不可复用的节点 ID；不为手工伪造的“同 ID、同 class 新节点”提供额外身份推断。
7. 原始对象中的显式 `null`、`false`、`0` 和空字符串都视为数据，不按真假值过滤。

### 4.3 导出规则

导出某个节点时按以下优先级构建 `port_defaultv`：

1. 若残留条目的 `class` 与当前导出 class 完全一致，先按原始字符串键恢复残留值。
2. 再把当前 schema 可映射端口的编辑值写入规范索引键。
3. 当前可编辑值优先于残留值。
4. class 不一致时忽略该节点的残留值，不把旧类型数据写入新类型。
5. 不对 `port_defaultv` 补齐中间索引，也不把对象转换成数组。

该设计只保存当前编辑器无法表达、但 legacy 文件已有的数据，不扩展可见节点端口，也不把未知字段提升为运行时输入。

### 4.4 Legacy 边身份与顺序

legacy `edges` 数组顺序必须作为数据保留，因为同一执行输出可能存在多条连接。迁移时 visible edge 和 hidden edge 会被拆开，若不记录顺序，导出时“先 visible、后 hidden”会改变原始交错顺序。

在 `GraphConnection` 增加仅用于 legacy 往返的可选 `legacyEdgeId` 和 `legacyOrdinal`；在 `GraphLegacyState` 增加与 `HiddenEdges` 等长的 `hiddenEdgeOrdinals`。序号使用指针或等价可空表示，确保原始序号 `0` 不被 `omitempty` 丢弃。前端 Rete connection 在加载、克隆和生成文档时透传这两个字段，不参与绘制和连线规则。

导出时：

1. 未经删除的 visible edge 复用原始 edge ID 和 ordinal；hidden edge 使用对应的 hidden ordinal。
2. 收集全部原始边后按 ordinal 稳定排序，恢复 visible/hidden 的原始交错顺序。
3. 新建连接没有 ordinal，按当前 GraphDocument 连接顺序追加在原始边之后，并生成新 edge ID。
4. ordinal 重复、越界或 `hiddenEdgeOrdinals` 与 `HiddenEdges` 长度不一致时显式报错，不能猜测顺序。
5. 用户删除可见节点时，同一个可撤销编辑事务同步删除所有引用该节点的 hidden edge 及其 ordinal，并在状态信息中报告额外清理数量；这样用户不会因不可见的悬空边而无法保存，也不会发生无提示清理。

旧版 native GraphDocument 可能已有 hidden edges 但没有 ordinal。此时按文档现有顺序给这些 hidden edges 分配追加序号并记录兼容性 warning；无法恢复从未保存过的历史交错顺序，但后续保存必须稳定。

## 5. Legacy 导出预检

legacy 导出改为失败关闭。开始构造输出前，完整检查：

- 最终导出的可见与 hidden 节点 ID 不为空，且整个集合中不重复。
- 每个可见节点都能映射到 legacy class 和端口规格。
- 每条可见连接的源节点和目标节点都存在且可映射。
- 每条连接的源、目标端口都能映射到 legacy 端口索引。
- 端口方向与连接类型满足 legacy 表达要求。
- 每条保留的 hidden edge 的两端都存在于最终导出的可见或 hidden 节点集合；不能继续用 `continue` 静默过滤。正常节点删除必须已按 4.4 同步清理关联 hidden edge。

任一条件失败时，导出返回包含节点 ID、连接数组索引、端口键和失败原因的错误。`GraphConnection` 当前没有连接 ID，错误信息不得虚构该字段。不得跳过对象后继续生成部分文件，也不得自动改存 native GraphDocument。

预检必须在写文件之前完成。前端保存流程在 `saveGraph` 内捕获错误，显示本地化的保存失败提示和后端明细，并保持标签页为未保存状态。因预检或 legacy 转换失败时不得进入 `os.WriteFile`，因此不会覆盖已有目标文件。通用文件写入的原子替换不在本轮范围内。

## 6. 编译器结构校验

### 6.1 Schema 版本

格式探测必须区分“字段不存在”和“字段存在但不受支持”：

- JSON 中存在 `schemaVersion` 字段时，只接受数值 `1`。
- 字段存在但值为 `0`、负数、大于 `1`、非整数或非数值时返回明确错误。
- 字段不存在时继续按现有 legacy `GraphConfig` 解析，不能破坏无版本号旧执行文件的兼容性。

`ParseGraphConfigJSON` 和 `parseGraphFile` 必须共用同一版本探测规则，避免目录加载与直接解析结果不一致。

### 6.2 节点与连接

编译阶段拒绝：

- 空节点 ID。
- 重复节点 ID。
- 空变量名和重复变量名。
- 不支持的变量类型，或无法转换成声明类型的非 nil 变量默认值；nil 继续表示使用该类型零值。
- 重复入口 ID，包括多个 `FunctionEntry`。
- 连接引用不存在的源或目标节点。
- 源输出端口索引越界或目标输入端口索引越界。
- 执行端口连接到数据端口，或数据端口连接到执行端口。
- 两端均为内建 `*Port` 且声明的数据类型不兼容。
- 同一个数据输入端口存在多个生产者。多个执行流可以汇入同一执行输入；是否为数据端口必须根据端口定义判断，不能只按节点和索引去重。

内建数据端口兼容矩阵与编辑器保持一致：同类型兼容，任一端为 `any` 时允许连接，其余组合不兼容。`any -> 具体类型` 在运行期按实际值转换，转换失败必须返回带节点和端口信息的执行错误；`具体类型 -> any` 保存实际值。对于自定义 `IPort`，编译器不猜测其类型系统，只执行接口层面可确认的方向和存在性校验；自定义端口的额外兼容性由其实现负责。

编译器不新增“图中存在任意环就失败”的规则。运行时输入绑定使用可返回错误的内部赋值函数，保留目标端口声明类型，不再通过 `SetValue` 把目标端口类型改成源端口类型。编译期可确定的不兼容直接失败，涉及 `any` 实际值的不兼容在运行期失败。

内部赋值规则必须固定为：nil 内建端口在编译期报错；同类型复制值且数组、`any` 容器执行深拷贝；具体类型写入 `any` 时克隆并保存源实际值；`any` 写入具体类型时使用该目标类型现有的 `setAnyValue` 转换规则并传播错误；执行端口不参与值复制。自定义 `IPort.SetValue` 保持接口自身语义，不假定其可返回转换错误。

### 6.3 已确认但延期的执行分支缺陷

历史 legacy 图允许同一个执行输出连接多个后继，仓库样本中已经存在这种结构。当前 `ExecNode.Next[output]` 只能保存一个后继并由后写覆盖先写，这是已确认的引擎缺陷，但不在本轮直接修改。

后续设计至少必须同时解决：每个分支独立的 activation/frame context；同步函数返回在父执行栈内调用 continuation 的重入路径；多个挂起分支恢复的串行调度；某分支报错或终止后其他 continuation 的失效；分支顺序和汇合语义。仅把 `Next` 改成二维切片或给 `Graph` 增加非重入互斥都会引入新 bug。

本轮编译器不得把同一执行输出多连接新增为错误，也不得宣称已经修复最后连接覆盖行为。legacy 持久化仍必须完整保留这些连接及原始顺序，为后续运行时修复提供正确输入。

## 7. 异步任务与实例释放

### 7.1 实例释放信号

每个运行实例持有只关闭一次的 release 信号、released 状态和执行 lease 计数。释放与执行采用明确的线性化规则：

- `Blueprint.Do` 在仍持有蓝图实例读锁时，从实例生命周期锁内取得一次 lease；取得失败则不启动执行。执行片段返回时释放 lease，包括返回 suspended 的情况。
- 所有挂起执行最终都通过 `Continuation.Resume` 恢复，因此 `Resume` 必须重新取得 lease，不能沿用挂起前的 lease，也不能只依赖 `Sleep` 或内建节点各自检查。
- `ReleaseGraph` 在生命周期锁内标记 released 后，新的 `Do`、`Resume` 不能取得 lease，尚未开始的 timer 回调也不能取得外部调用资格。
- 释放先取得线性化点时，`Resume` 返回明确的 released 错误且不执行后续节点；`Resume` 先取得 lease 时，视为释放前已开始的在途片段，可以在锁外完成。
- `ReleaseGraph` 不等待 lease 计数归零，避免从蓝图执行链或重入回调内释放自身时死锁。其保证是“返回后不能开始新片段”，不是强行中断已经在执行的 Go 调用。
- lease 只覆盖实际执行片段，不跨异步挂起期长期持有；最后一个 lease 归还后实例运行资源可自然释放。
- 没有关联 `GraphInstance` 的独立 `Graph` 保持现有 continuation 行为。
- `Sleep` 仍监听 release 信号并停止自己的 `time.Timer`，用于及时释放定时器资源；正确性最终由 `Continuation.Resume` 的中央检查兜底。

### 7.2 外部定时器

实例维护自己创建的 timer 登记项。每个登记项使用实例级单调递增的内部 token，避免多个并发 `Do` 各自从 `1` 计数导致冲突；登记项可在绑定后保存外部 timer ID 或本地 `time.Timer` 取消句柄。注册、正常触发消费和释放清空必须使用同一生命周期/定时器锁：

- 调用外部 `SafeAfterFunc` 前，先在锁内创建 pending 登记项；外部调用和回调均通过内部 token 引用该项。
- `SafeAfterFunc` 返回外部 ID 后，在锁内完成绑定并建立外部 ID 到 token 的索引。若登记项已被回调消费或实例已经释放，则不再绑定，并在锁外立即取消返回的外部 ID。
- 外部实现允许零延时甚至同步触发回调；回调可在 ID 绑定前按 token 消费 pending 项，不能依赖“调用返回后再登记”的时序。
- 无 module 的本地 timer 在实例锁内分配实例级公开 ID，创建后把 `*time.Timer` 绑定到同一登记项；若绑定时已释放，立即在锁外停止 timer。
- 正常触发时在锁内消费登记项并取得一次“允许调用外部 timer callback”的资格，再解锁调用模块。该资格不是蓝图执行 lease。
- 手工取消也先在锁内消费登记项，再在锁外调用外部模块或本地取消句柄。
- 释放先标记实例失效并清空集合后，新的注册或回调不能再取得资格。
- 回调先取得外部调用资格时视为释放前已经开始的在途回调，可在锁外完成；重复回调不能再次进入蓝图。

外部 timer callback 调用 `module.TriggerEvent` 后，真正的蓝图入口仍通过公开 `Blueprint.Do` 重新取得执行 lease；不跨模块边界传递或借用 callback 资格。若 `ReleaseGraph` 恰好在线程从 callback 进入 `Do` 之前线性化，外部 callback 可以完成，但蓝图事件必须被拒绝。这一竞态是“释放后不开始新执行片段”的预期行为。

登记项状态至少区分 pending、bound、fired、cancelled 和 released。active registry 消费登记项后，创建方和回调仍共享一个小型状态对象作为 tombstone，直到 `SafeAfterFunc` 返回并完成 ID 绑定判定；随后才允许释放。返回时若 tombstone 已是 fired，不做第二次取消；只有 cancelled 或 released 且外部 ID 尚未绑定的项在绑定后补做一次锁外取消。fired 路径由 callback 执行唯一一次既有外部清理。外部取消接口按可能非幂等处理，不能依赖重复调用无害。legacy `cancelTimer` 回调继续用于原有兼容路径，本地 `time.Timer` 始终直接停止自己的句柄。

释放实例时：

1. 在蓝图实例锁内移除实例。
2. 在实例生命周期/定时器锁内标记 released、关闭 release 信号，并取出及清空全部外部和本地定时器。
3. 释放所有内部锁。
4. 在锁外调用外部模块或本地取消句柄。

无外部模块时创建的本地定时器也必须登记到实例生命周期中，确保后续取消和实例释放都能找到同一个定时器，而不是只存在于单次执行会话。

### 7.3 锁顺序与回调边界

- 蓝图实例锁保护实例集合，以及运行状态快照指针的替换。
- 每份运行状态自己的变量锁只保护该状态的变量映射读取、克隆和修改。
- 实例生命周期/定时器锁保护 released 状态、release 信号、lease 计数、timer token 序列和定时器登记表，并作为执行片段资格、timer callback 资格与释放的线性化点。
- 热加载需要同时持锁时固定使用“蓝图实例锁，再到旧状态变量锁”的顺序；任何变量访问路径都不得反向获取蓝图实例锁。
- 不在持有上述任一锁时调用外部模块、用户回调或 continuation。
- 需要跨资源操作时先在各自锁内提取不可变工作项，解锁后执行外部动作。

该约束避免外部模块重入蓝图时产生锁反转或死锁。

## 8. 热加载变量快照

每个顶层实例把 `compiled`、`variables` 和对应 `variableMu` 封装为一个运行状态对象。`Do` 在蓝图实例读锁内只捕获一次状态对象指针，后续整条同步或异步执行链都使用该对象。热加载采用复制后交换：

1. 编译新 GraphDocument，失败时保留旧编译结果和旧实例状态。
2. 对每个实例构建新的顶层变量映射。
3. 名称相同且规范化后的端口类型相同的变量，深克隆保留当前值；类型字符串大小写或兼容别名不能导致无意义重置。
4. 新增变量使用新文档中的默认值。
5. 类型变化的变量重置为新文档默认值。
6. 已删除变量不进入新映射。
7. 应用热加载时先持有蓝图实例锁，再逐个取得旧状态变量锁，克隆当前值并把实例状态指针一次替换为包含新编译结果、新变量映射和新变量锁的新状态对象。不得在锁内调用外部代码。

一次执行会话在开始时捕获运行状态对象。已经开始或挂起中的会话继续使用旧状态，新会话使用新状态，禁止在一条执行链中途混用新旧节点表或变量表。热加载交换后，旧会话对旧变量的后续写入只属于旧会话，不反向合并到新状态；旧状态由仍持有它的会话自然释放。

蓝图实例锁与旧状态变量锁共同构成热加载线性化边界：在取得旧变量锁之前已经完成的变量写入必须进入新状态；交换后仍持有旧状态的在途会话，其后续写入只留在旧状态。若新加载集合删除了某个 graph，已有实例保持原来的 compiled 和变量状态并计入 `UnchangedInstances`，但后续 `Create` 不能再创建该 graph；本轮不隐式释放仍在运行的实例。

这里的迁移只适用于长期存在的顶层蓝图实例变量，不适用于函数局部变量。

## 9. 函数局部变量语义

函数蓝图中的变量遵循调用帧语义：

- 每次调用都从函数蓝图变量定义和默认值创建全新的局部变量映射。
- 上一次调用对局部变量的修改不能影响下一次调用。
- 递归调用的每一层都有独立局部变量映射。
- 函数局部变量与调用方蓝图中的同名变量互不影响。
- 函数挂起后恢复时继续使用该次调用自己的局部变量映射。
- 函数返回后，该调用帧不再被顶层实例或后续调用引用。

函数子图使用自己的变量锁，不共享调用方实例的变量锁。函数参数和返回值只通过既有入口、出口端口传递，不通过同名变量隐式共享。

维护文档中“函数调用共享实例变量”的描述必须改为上述规则。

## 10. 测试策略

### 10.1 Legacy 往返

- 提交一组最小、脱敏的 legacy fixture，覆盖本次发现的默认值残留类型、未知节点/边和多执行分支；CI 自动测试只能依赖这些受 Git 管理的 fixture。
- 当前本机 `build/bin/vgf` 中识别出的 158 个 legacy `.vgf` 不受 Git 管理，不能作为 fresh clone 的必需测试输入。提供可选的外部样本审计命令或测试参数：路径存在时递归执行全部样本，路径不存在时明确报告跳过，不把跳过伪装成通过。
- 做规范化后的语义比较：节点相同，原有连接的顺序、端点和 `edge_id` 相同，所有 `port_defaultv` 原始键和值相同；只忽略 JSON 对象键顺序。
- 覆盖 `null`、`false`、`0`、空字符串、越界索引、负数键、非数字键和非规范数字键。
- 增加节点 ID 复用但 class 改变时不恢复残留值的测试。
- 增加未知节点、未知端口和不可映射连接导出失败且不生成部分文件的测试。
- 增加 hidden edge 端点缺失时显式失败、不得静默过滤的测试。
- 增加 visible/hidden edge 原始交错顺序往返测试，以及删除可见节点时同步清理关联 hidden edge/ordinal 的前端测试。
- 增加前端保存失败后标签仍为 dirty、原文件未被覆盖的测试，同时覆盖已有 legacy 文件直接保存和“另存为 `.vgf`”两条路径。

### 10.2 编译器

- 覆盖 schema 字段缺失的 legacy GraphConfig 仍可解析，以及字段存在但版本不支持时失败。
- 覆盖空/重复节点 ID、空/重复变量名、非法变量类型/默认值、重复入口 ID、重复数据输入生产者、端口越界、方向错误和内建类型不匹配。
- 覆盖 `any` 双向连接；实际值可转换时成功，不可转换时返回执行错误且目标端口声明类型不变。
- 覆盖合法流程回边仍可编译。
- 验证类型错误不会改变目标端口声明类型。

### 10.3 生命周期

- `Sleep` 到期前释放实例，后续节点不得执行。
- 自定义异步节点直接持有 continuation，实例释放后调用 `Resume` 也不得执行后续节点。
- `Do` 或 `Resume` 在释放前取得 lease 时允许完成；释放线性化后不能取得新 lease，`ReleaseGraph` 从执行链内部调用也不死锁。
- 外部定时器正常触发后从实例集合移除。
- 实例释放取消仍登记的外部和本地定时器。
- 并发注册与释放时不存在释放后登记的 timer。
- 外部 `SafeAfterFunc` 同步触发回调时不会漏登记、重复触发或死锁。
- timer callback 在释放前取得外部调用资格、但 `TriggerEvent` 对应的 `Do` 晚于释放时，callback 可返回而蓝图入口不执行。
- pending 登记被 fired、cancelled、released 三种路径消费后，外部 ID 绑定与清理都恰好一次，tombstone 最终可释放。
- 多个并发 `Do` 创建本地 timer 时公开 ID 唯一，并能分别取消正确的 `*time.Timer`。
- 定时器触发与释放并发执行，多次重复运行并通过 race 检查。
- 回调重入蓝图不发生死锁。

### 10.4 热加载

- 同名同类型变量保留当前值。
- 新变量获得默认值，删除变量消失，改类型变量重置。
- 编译失败保持旧状态。
- 挂起中的旧会话使用旧状态，热加载后的新会话使用新状态；旧会话交换后的变量写入不污染新状态。
- 新加载集合删除某个 graph 后，已有实例保留旧状态，新建实例找不到该 graph。

### 10.5 函数局部变量

- 连续两次调用时，第二次从默认局部变量开始。
- 调用方和函数存在同名变量时互不影响。
- 两层递归调用变量隔离。
- 函数经过异步挂起再恢复时保留本调用局部状态。
- 函数返回后再次调用仍从默认局部状态开始，顶层实例结构不保存函数调用帧。

当前全量测试中依赖缺失业务 schema 的三个样本测试应单独记录，不通过放宽编译、迁移或导出规则来掩盖。若这些测试在实现前仍失败，报告必须区分既有环境问题与本轮回归。

## 12. 风险与缓解

- **旧文件残留数据污染新节点**：残留值同时绑定节点 ID 和 legacy class，class 变化时不恢复。
- **导出规则变严格影响现有工作流**：错误包含具体对象和原因，预检失败保持 dirty 且不进入文件写入；不再生成表面成功的残缺文件。
- **锁调整引入死锁**：内部锁只用于提取或交换状态，所有外部调用和 continuation 都在锁外；增加重入和 race 测试。
- **释放与回调同时发生时语义不清**：用生命周期锁定义线性化点；释放前取得资格的在途执行允许完成，释放后不能取得新资格。
- **外部 timer 在返回 ID 前同步回调**：先登记内部 token，再调用外部模块；绑定、回调、取消和释放都按 token 协调。
- **热加载混用新旧状态**：执行会话捕获单一状态对象，状态指针交换发生在事务边界。
- **热加载后旧会话写入丢失**：这是快照边界的明确语义；旧写入只影响旧状态，不覆盖已经发布的新变量状态。
- **函数变量被误并入实例迁移**：顶层实例变量与函数调用帧使用不同创建路径和测试矩阵。
- **严格类型校验拒绝合法 `any` 连接**：兼容矩阵与编辑器一致，`any` 的实际值转换错误留到运行期返回；自定义 `IPort` 不做类型推断。
- **legacy 执行分支被误判或被不完整修复**：本轮只保证持久化不丢边、不改顺序且编译器不新增拒绝；运行语义留给独立的 activation/frame 设计。
- **本地样本未纳入版本控制**：CI 使用已提交脱敏 fixture，本机完整样本只作为显式报告输入的扩展审计。
- **合法循环被误判**：本轮不做全图无环校验，只校验连接结构和端口契约。

## 13. 实施顺序

1. 先用失败测试固定 legacy 默认值丢失、边顺序变化和导出静默跳过行为，再实现残留数据、边身份/ordinal 透传、节点删除清理和预检。
2. 增加格式探测和编译器失败测试，再补版本、结构、数据输入唯一性和端口兼容校验。
3. 增加释放后恢复、并发定时器注册/消费和重入测试，再调整中央 continuation 与实例生命周期。
4. 增加热加载变量迁移与会话状态测试，再引入运行状态对象并实现复制后交换。
5. 增加函数局部变量隔离回归测试，移除父级变量锁共享并修正文档。
6. 运行窄测试、race、重复测试、全量测试、前端测试、样本往返审计和基准对比。

每一步保持可独立验证，不把文件格式、并发和函数语义改动合并成一个不可定位的提交。
